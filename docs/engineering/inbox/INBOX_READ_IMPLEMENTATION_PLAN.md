# Inbox Read Implementation Plan

**Status:** Historical migration plan

This document is retained to explain how the canonical Inbox Read model was planned and rolled out. It is not the current runtime source of truth.

For current behavior and ownership, use:

- `READ_UNREAD_BEHAVIOR.md`
- `INBOX_READ_ARCHITECTURE.md`
- `INBOX_GROUPING_V2_ARCHITECTURE.md`

Source documents:

- `docs/engineering/inbox/READ_UNREAD_BEHAVIOR.md`
- `docs/engineering/inbox/INBOX_READ_ARCHITECTURE.md`

This plan converts the approved architecture into a practical implementation roadmap. It does not redefine Inbox product behavior, required contexts, or activity semantics.

# Overview

The implementation should move the Inbox from a derived, timestamp-based read model to the approved canonical activity model in small, reviewable pull requests.

Current implementation:

- Inbox rows are derived at query time from `mark_events`, current `mentions`, and current workspace invite facts in `apps/web/src/lib/workspace/inbox-query.ts`.
- Unread state is derived from one workspace/user timestamp in `inbox_read_states.lastReadAt`.
- `markInboxReadAction()` is the only normal read-state write path and is used by "Mark all read".
- Dashboard navigation and Mark detail rendering do not currently acknowledge activity-specific context views.

Transitional implementation:

- Add `inbox_activities` and `inbox_activity_read_states` without changing the existing Inbox UI.
- Backfill and dual-write canonical activity rows while the legacy derived Inbox remains available.
- Convert read writes to per-activity read-state rows while optionally continuing to update `inbox_read_states` for rollback.
- Switch the Inbox projection only after canonical activity counts and read states have been validated.

Final implementation:

- `inbox_activities` is the durable, append-only activity layer.
- `inbox_activity_read_states` is the mutable acknowledgement layer.
- `loadInboxSnapshotForWorkspace()` reads canonical activities and calculates unread from missing read-state rows.
- "Mark all read" inserts read-state rows for currently unread activities.
- Automatic read acknowledgements run only after the required context has actually rendered and become visible.
- Legacy timestamp unread logic is removed after the migration window closes.

Incremental delivery is preferred because this feature crosses schema, RLS, activity generation, read projection, navigation, realtime invalidation, and client visibility detection. Keeping each phase independently shippable lets engineers validate one boundary at a time and revert safely without breaking the existing Inbox.

# Guiding Principles

- Keep the application working after every phase.
- Avoid breaking the existing Inbox while canonical data is being introduced.
- Prefer additive migrations until the final cleanup phase.
- Preserve the approved product behavior exactly.
- Keep old and new read models comparable during migration.
- Make every phase independently testable.
- Avoid large rewrites of Inbox UI, dashboard detail UI, or workspace bootstrap data.
- Treat activity facts as immutable and read acknowledgements as mutable.
- Use idempotent writes for backfill, dual-write, and context-viewed acknowledgement.
- Keep RLS and recipient ownership constraints in place before exposing new tables to client-side realtime.

# Phase 1 - Add Canonical Tables Behind Existing Behavior

## Goal

Add the target storage model without changing user-facing Inbox behavior.

Why it exists:

- The approved architecture requires a durable activity layer and a separate read-state layer.
- This phase creates the database and Drizzle foundation while keeping the current timestamp implementation active.

Dependencies:

- Approved architecture option 4: `inbox_activities` plus `inbox_activity_read_states`.
- Existing workspace membership and recipient ownership rules.

## Files expected to change

- `apps/web/src/db/schema.ts`
- `apps/web/supabase/setup.sql`
- `apps/web/supabase/rls-and-auth.sql`, if this remains the RLS source file for local setup
- `apps/web/drizzle/<generated>_inbox_activities.sql`
- `apps/web/drizzle/meta/_journal.json`
- Drizzle snapshot metadata generated by the implementation PR

## Database changes

- Add `inbox_activities` as an append-only table.
- Add `inbox_activity_read_states` as the mutable per-user read-state table.
- Add indexes for recipient timeline loading, unread lookup, source deduplication, and required-context lookup.
- Add RLS policies so users can read only activities where they are the recipient and can write read state only for their own activities.
- Keep `inbox_read_states` unchanged.
- Do not add realtime publication changes yet unless the implementation PR can verify them independently.

## Application changes

- Add Drizzle table definitions, relations, and inferred types.
- Add enum or text-type declarations needed to represent the approved activity and required-context contracts.
- Do not update `loadInboxSnapshotForWorkspace()`.
- Do not update `markInboxReadAction()`.
- Do not update Inbox UI.

## Testing

- Run schema type checks.
- Validate generated SQL locally against a clean database.
- Verify RLS policy shape for owner/member recipient access.
- Confirm existing Inbox tests and build behavior still use `inbox_read_states`.

## Success criteria

- The application works exactly as it did before the phase.
- New tables exist with indexes, constraints, and RLS.
- Existing Inbox queries and "Mark all read" behavior are unchanged.
- The migration is additive and can be deployed before any application code starts using the tables.

# Phase 2 - Build Canonical Projection and Backfill Helpers

## Goal

Create server-side helpers that can convert current activity sources into canonical activity rows, without switching the Inbox reader.

Why it exists:

- Backfill and dual-write need one shared mapping path so source identity, required context, payload, and recipient rules are consistent.
- The architecture explicitly allows migration from current facts before replacing the derived read model.

Dependencies:

- Phase 1 schema.
- Current source loaders in `apps/web/src/lib/workspace/inbox-query.ts`.
- Current mention synchronizer in `apps/web/src/lib/workspace/mentions.ts`.

## Files expected to change

- `apps/web/src/lib/workspace/inbox-model.ts`
- `apps/web/src/lib/workspace/inbox-query.ts`
- New helper module under `apps/web/src/lib/workspace/` for canonical activity projection
- New test file under `apps/web/src/lib/workspace/` for projection and deduplication

## Database changes

- None beyond Phase 1.
- Backfill helpers should write to the new tables only when explicitly invoked by an implementation script, action, or migration step.

## Application changes

- Add mapping helpers for current product-mapped sources:
  - mark-event facts that map to Assignment, Comment, Workflow Change, Status Change, Priority Change, and Label Change.
  - mention facts that map to Mention.
- Preserve current recipient behavior for mark-event activity during migration:
  - marks assigned to the viewer.
  - marks where the viewer has authored at least one comment.
  - events authored by the viewer are excluded for that viewer.
- Store deterministic source keys so repeated backfills do not create duplicates.
- Store required context type and id on every canonical activity row.
- Store immutable payload needed for display fallback.
- Keep current non-product-mapped events, such as `prompt_copied`, out of final read semantics unless the approved source documents are updated.

## Testing

- Unit test source key generation.
- Unit test required-context mapping for each approved activity type covered by current data.
- Unit test that self-authored events do not create recipient activities for the actor.
- Unit test that repeated projection is idempotent.
- Add a comparison test that canonical projection produces the expected activity count for representative current Inbox inputs.

## Success criteria

- Canonical activity rows can be produced from current facts in a deterministic way.
- No production UI reads canonical rows yet.
- Existing Inbox behavior remains unchanged.
- Engineers can safely backfill a workspace more than once without duplicate activities.

# Migration Strategy

The old and new implementations should coexist through Phases 1 through 6.

Current implementation:

- Legacy Inbox projection derives activity identity from `mark_events`, current `mentions`, and invite facts.
- Legacy unread state uses `inbox_read_states.lastReadAt`.

Transitional implementation:

- Canonical tables are added additively.
- Backfill creates canonical activity rows from reconstructable existing facts.
- Dual-write keeps canonical activity rows current for new facts.
- "Mark all read" writes per-activity read-state rows and may continue updating `inbox_read_states.lastReadAt` for rollback.
- The canonical reader can be compared against the legacy reader before becoming user-facing.

Final implementation:

- Canonical activities and read states are the only Inbox read model.
- Automatic context-viewed acknowledgement is wired for every approved activity type.
- `inbox_read_states` and timestamp unread logic are removed or left unused after the rollback window.

Old code can safely be removed when:

- All Inbox readers use canonical activities.
- All approved activity producers write canonical rows.
- Read acknowledgements no longer depend on `lastReadAt`.
- Realtime invalidation covers canonical activity and read-state changes.
- A production observation window shows no meaningful divergence between expected and actual Inbox counts.

Do not remove `mark_events`, `mentions`, invites, or review-link source tables as part of this migration. They remain source facts or feature data; only the legacy Inbox read-state model is retired.

# Rollback Strategy

Phase 1 rollback:

- Because schema changes are additive, leave the new tables unused or drop them in a follow-up migration if no later phase has shipped.
- Application rollback is not needed if no code reads or writes the tables.

Phase 2 rollback:

- Disable or remove projection helpers.
- Existing user-facing Inbox remains on the legacy reader.
- Canonical rows created in local or staging validation can be truncated if needed.

Phase 3 rollback:

- Stop running the backfill.
- Keep already backfilled rows unused.
- Rerun backfill later because source keys are deterministic and idempotent.

Phase 4 rollback:

- Disable canonical dual-write producers.
- Keep legacy source writes and legacy Inbox reader active.
- Existing canonical rows remain harmless because they are not yet the user-facing source.

Phase 5 rollback:

- Revert "Mark all read" to timestamp-only writes.
- Continue using `inbox_read_states.lastReadAt` while the canonical read-state rows remain unused.
- If both write paths shipped, leave per-activity read states in place for future retry.

Phase 6 rollback:

- Switch `loadInboxSnapshotForWorkspace()` back to the legacy derived loader.
- Continue dual-writing canonical rows so the migration can be retried after fixing the issue.

Phase 7 rollback:

- Remove new realtime handlers or ignore canonical-table events.
- Fall back to explicit refetch and legacy broad workspace invalidation while investigating.

Phase 8 and Phase 9 rollback:

- Disable automatic context-viewed client calls.
- Keep "Mark all read" available as the convenience fallback.
- Leave read-state rows already written; they represent acknowledged activity and should not be mass-deleted unless there is confirmed corruption.

Phase 10 rollback:

- Disable the new Invite or Review Link producers and context acknowledgements.
- Keep existing invitation and review-link workflows active.
- Keep canonical rows unused or hidden if the source mapping needs correction.

Phase 11 rollback:

- Do not start this phase until the rollback window for earlier phases has closed.
- If rollback is still required, retain `inbox_read_states` and legacy loader code until canonical production behavior has been stable.

# Risks

- Backfilled recipients may not perfectly match the audience at the time of historical `mark_events`.
  - Mitigation: preserve current recipient rules during migration and measure divergence before switching reads.
- Historical mention activities cannot be reconstructed after current `mentions` rows were deleted.
  - Mitigation: document this as a backfill limitation and rely on dual-write for future immutability.
- Duplicate activities could be created by backfill plus dual-write.
  - Mitigation: enforce deterministic source keys and database uniqueness.
- Automatic context acknowledgement could mark read too early.
  - Mitigation: acknowledge only after rendered context exists and visibility checks pass.
- Automatic context acknowledgement could fail for collapsed, virtualized, or delayed UI.
  - Mitigation: test target visibility paths and keep "Mark all read" available.
- Realtime filters could over-invalidate or leak activity existence.
  - Mitigation: keep RLS recipient-scoped and route client invalidation by current user/workspace.
- Bulk "Mark all read" could become expensive for large unread sets.
  - Mitigation: use set-based inserts, batching if needed, and indexes for unread lookup.
- Current non-product-mapped events may appear in legacy Inbox but not have approved final read semantics.
  - Mitigation: do not assign final required-context behavior without updating the source documents.
- Deleted source data could make activity display incomplete.
  - Mitigation: store immutable payload and use it as fallback.
- Rollback after legacy removal would be difficult.
  - Mitigation: keep Phase 11 separate and delay it until production stability is proven.
